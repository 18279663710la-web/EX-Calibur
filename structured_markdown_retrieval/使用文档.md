## 系统概述
来源文件：使用文档.md
文档名称：使用文档

检索关键词：使用文档 使用文档.md 系统概述


CloudRAG-Hub 是一个**云原生架构的多模态智能文件检索与知识库管理系统**，面向高校计算机科学学院设计。核心能力：

| 能力 | 说明 |
|------|------|
| **文件管理** | 拖拽上传 PDF/DOCX/MD/图片等 12 种格式，自动向量化处理 |
| **知识库检索** | 基于 Dify RAG 引擎的混合检索（关键词 + 向量 + Rerank） |
| **流式对话** | SSE 逐 Token 打字机效果，Markdown 渲染，文档引用溯源 |
| **数据看板** | Token 消耗、延迟聚合、用户排行、模型分布、成本估算 |
| **权限体系** | JWT 双 Token 认证，用户/管理员角色隔离 |
| **审计追踪** | 全量操作日志，每次 API 调用记录延迟和 Token 使用 |
| **云原生部署** | Docker Compose 一键编排 3 个服务，非 root 运行，健康检查 |

**数据流（一次知识库对话请求的完整路径）：**

```
1. 用户输入问题 → 2. Svelte 封装 ChatRequest → 3. Fetch POST /api/v1/knowledge-base/chat
→ 4. Nginx 反向代理到 backend:8080 → 5. JWT 中间件验证
→ 6. 审计日志记录请求 → 7. 写入 user message 到 PostgreSQL
→ 8. POST Dify /chat-messages (SSE streaming, API Key 隐藏) → 9. Dify 工作流执行
→ 10. 知识检索节点查询向量库 → 11. LLM 节点生成回答(逐 token)
→ 12. Dify SSE 流回到后端 → 13. 后端封装为 CloudRAG 标准 SSE → 14. Nginx 透传
→ 15. 前端 ReadableStream 解析 → 16. 逐 token 追加到 DOM (打字机) → 17. done 事件渲染 Markdown
```

### 使用 Docker Compose（推荐）

这是最快的方式，只需 4 步：

**步骤 1：克隆或解压项目**

```bash
cd CloudRAG-Hub
```

**步骤 2：配置环境变量**

编辑根目录 `.env` 文件，填入 Dify API Key（必填）：

```env
# 必填：你的 Dify 应用 API Key
DIFY_API_KEY=app-xxxxxxxxxxxxxxxxxxxxxxxx

# 推荐修改：
JWT_SECRET_KEY=生成一个64字符随机字符串
POSTGRES_PASSWORD=设置数据库密码
```

> 如果还没有 Dify API Key，参见 [第 8 节](#8-dify-ai-平台集成) 先创建 Dify 应用。

**步骤 3：一键启动**

```bash
docker-compose up -d
```

等待 30 秒左右，所有服务健康检查通过后即可访问。

**步骤 4：验证**

```bash
# 查看服务状态
docker-compose ps

# 应该看到 3 个服务都是 healthy 状态
# cloudrag-postgres   healthy
# cloudrag-backend    healthy
# cloudrag-frontend   healthy
```

打开浏览器访问 `http://localhost`。

#### Dify SSE 代理（`services/dify.py`）

```python
# 这是整个后端最核心的代码逻辑，流程图：

# 1. 接收前端 ChatRequest
# 2. 创建/获取 conversation → 写入 user message 到 PostgreSQL
# 3. 发送 SSE meta 事件（conversation_id, model）
# 4. 发送 SSE references 事件（检索到的文档片段）
# 5. 调用 Dify /chat-messages API（response_mode=streaming）
#    - 隐藏 DIFY_API_KEY（只在后端内存中，从不返回前端）
#    - 流式读取 Dify 的 SSE 事件

#    - 持久化 assistant message（content + tokens + latency_ms）
#    - 更新 conversation.updated_at
#    - 发送 SSE done 事件
```

### 独立开发模式（无需后端）

前端内置了完整的 Mock 引擎，无需后端即可体验所有功能：

- **登录**：输入任意用户名密码可直接「登录」（Mock 模式下自动创建假 Token）
- **对话**：点击发送会触发 `createMockChatStream`，模拟完整的 SSE 打字机流
- **文件管理**：展示空状态界面（可测 UI 交互）
- **数据看板**：需要管理员角色（Mock 模式下 admin 用户可进入）

要启用真实后端，只需关闭 Mock 模式：编辑 `ChatView.svelte`，将 `useMock` 改为 `false`。

### 前端 API 层使用示例

```typescript
// 1. 导入
import { authApi, filesApi, dashboardApi, kbApi, saveAuth, clearAuth } from './lib/api';

// 2. 登录
const res = await authApi.login({ username: 'admin', password: 'Admin@123456' });
saveAuth(res.data!);  // 保存 Token 到 localStorage

// 3. 上传文件
const file = document.querySelector('input[type=file]')!.files![0];
const uploadRes = await filesApi.upload(file, 'AI,论文', '云计算相关论文');

// 4. 获取文件列表
const listRes = await filesApi.list({ keyword: '论文', status: 'ready', page: 1 });

// 5. SSE 对话流
import { createChatStream } from './lib/sse';
const { abort } = createChatStream(
  {
    conversation_id: null,
    query: '边缘计算是什么？',
    file_ids: [],
    model: 'gpt-4o',
  },
  (state) => {
    console.log(state.content);   // 累积的文本
    console.log(state.isStreaming); // 是否还在流式推送
  }
);

// 6. 停止流式
abort();
```

## Dify AI 平台集成
来源文件：使用文档.md
文档名称：使用文档
章节标题：Dify AI 平台集成
检索关键词：使用文档 使用文档.md Dify AI 平台集成 Dify AI 平台集成


### 概述

CloudRAG-Hub 依赖 Dify 作为 AI 编排引擎。你需要一个 Dify 账号并创建一个 Chatflow 应用。详细步骤见 `dify-setup-guide.md`，以下是操作摘要：

### 创建 Dify 应用（摘要流程）

| 步骤 | 操作 | 关键配置 |
|------|------|----------|
| 1 | 注册 [cloud.dify.ai](https://cloud.dify.ai) | — |
| 2 | 创建知识库 | 上传测试文档，等待向量化完成 |
| 3 | 创建 Chatflow 应用 | `[Start] → [知识检索] → [LLM] → [Answer]` |
| 4 | 配置知识检索节点 | 混合检索、Top K=5、阈值=0.7、Rerank 开启 |
| 5 | 配置 LLM 节点 | System Prompt 中注入 `{{#知识检索.result#}}`，开启对话记忆 |
| 6 | 发布应用 | 获取 `app-xxx` 格式的 API Key |
| 7 | 填入 `DIFY_API_KEY` | 后端 `.env` 或根目录 `.env` |

### Dify 输入变量映射

| CloudRAG-Hub `ChatRequest` 字段 | Dify `inputs` 变量 | 说明 |
|------|------|------|
| `query` | `sys.query` | 用户问题（系统变量） |

## 12. SSE 流式对话集成指南
来源文件：使用文档.md
文档名称：使用文档
章节标题：12. SSE 流式对话集成指南
检索关键词：使用文档 使用文档.md 12. SSE 流式对话集成指南 12. SSE 流式对话集成指南


> 这一节是给**其他前端开发者**在对接 SSE 接口时的参考。

### 12.1 协议概述

CloudRAG-Hub 的 SSE 流遵循标准 `text/event-stream` 协议。每个事件由 `event:` 行指定类型，`data:` 行携带 JSON 载荷，空行分隔。

### 12.2 事件生命周期

```
POST /api/v1/knowledge-base/chat
    │
    ▼
[meta] ─────── 对话元信息（conversation_id, model, ...）
    │
    ▼
[references] ─── 一次性的文档引用列表
    │
    ▼
[message] × N ── 逐 Token 增量文本（核心事件，打字机效果）
    │
    ├── [tool_call] × M ── 可选的工具调用事件
    │
    ▼
[done] ──────── 流结束，包含 token 统计和延迟
  ┃
  ┃  异常路径：
  ┃
[error] ──────── 流异常终止
```

### 12.3 JavaScript 接收示例

```typescript
// 使用 fetch + ReadableStream（推荐，支持 POST + 自定义请求体）
async function chatWithSSE(query: string, token: string) {
  const response = await fetch('/api/v1/knowledge-base/chat', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${token}`,
      'Accept': 'text/event-stream',
    },
    body: JSON.stringify({
      conversation_id: null,
      query: query,
      file_ids: [],
      model: 'gpt-4o',
      retrieval_config: { top_k: 5, score_threshold: 0.7, rerank_enabled: true },
    }),
  });

  const reader = response.body!.getReader();
  const decoder = new TextDecoder();
  let buffer = '';
  let currentEvent = '';
  let fullContent = '';

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;

    buffer += decoder.decode(value, { stream: true });
    const lines = buffer.split('\n');
    buffer = lines.pop() || '';

    for (const line of lines) {
      if (line.startsWith('event: ')) {
        currentEvent = line.slice(7).trim();
      } else if (line.startsWith('data: ')) {
        const data = JSON.parse(line.slice(6));

        switch (currentEvent) {
          case 'message':
            // ★ 逐 token 追加到 UI
            fullContent += data.token;
            appendToUI(data.token);  // 追加到 DOM
            break;
          case 'references':
            showReferences(data.chunks);
            break;
          case 'done':
            renderMarkdown(fullContent);
            showStats(data.usage, data.latency_ms);
            break;
          case 'error':
            showError(data.message);
            break;
        }
      }
    }
  }
}
```

### 12.4 Python 接收示例

```python
import httpx
import json

async def chat_stream(query: str, token: str):
    async with httpx.AsyncClient(timeout=120) as client:
        async with client.stream(
            "POST",
            "http://localhost:8080/api/v1/knowledge-base/chat",
            headers={
                "Authorization": f"Bearer {token}",
                "Content-Type": "application/json",
                "Accept": "text/event-stream",
            },
            json={"conversation_id": None, "query": query, "file_ids": [], "model": "gpt-4o"},
        ) as response:
            current_event = ""
            async for line in response.aiter_lines():
                if line.startswith("event: "):
                    current_event = line[7:].strip()
                elif line.startswith("data: "):
                    data = json.loads(line[6:])
                    if current_event == "message":
                        print(data["token"], end="", flush=True)  # 打字机效果
                    elif current_event == "done":
                        print(f"\n\n--- Tokens: {data['usage']['total_tokens']} ---")
```

### Q2：后端日志显示 `502 AI 服务不可用`

**原因：** Dify API Key 未配置或无效。

**解决：**

```bash
# 检查环境变量
docker exec cloudrag-backend env | grep DIFY

# 如果 DIFY_API_KEY 为空，编辑 .env 填入正确值后重启

  -d '{"inputs":{},"query":"测试","response_mode":"streaming","user":"test"}'
```

### Q3：SSE 对话流在中途断开

**原因：** Nginx `proxy_read_timeout` 过短或网络不稳。

**解决：** 检查 `nginx.conf` 中 `proxy_read_timeout` 是否设为 600s。

### Q4：文件上传提示「配额不足」

**原因：** 用户已用光 1GB 存储配额。

**解决：**

```sql
-- 为某个用户增加配额
UPDATE users SET quota_total_bytes = 10737418240 WHERE username = 'admin';
-- 或者重置已用配额
UPDATE users SET quota_used_bytes = 0 WHERE username = 'demo_user';
```

### Q5：数据库连接失败

**原因：** PostgreSQL 容器未就绪或凭证不匹配。

**解决：**

```bash
# 检查 PostgreSQL 是否 healthy
docker-compose ps postgres

# 手动测试连接
docker exec cloudrag-postgres psql -U cloudrag -d cloudrag -c "SELECT 1"

# 检查 backend 的 POSTGRES_* 环境变量是否与 postgres 服务一致
docker exec cloudrag-backend env | grep POSTGRES
```

### Q6：前端页面空白或 404

**原因：** Vite 开发模式 OK 但生产构建出现问题。

**解决：**

```bash
# 检查 Nginx 是否正常
docker exec cloudrag-frontend nginx -t

# 检查静态文件是否存在
docker exec cloudrag-frontend ls -la /usr/share/nginx/html/

# 查看 Nginx 错误日志
docker exec cloudrag-frontend cat /var/log/nginx/error.log
```

### Q7：JWT Token 总是过期

**原因：** `ACCESS_TOKEN_EXPIRE_SECONDS` 设置过短。

**解决：** 编辑 `.env`，增大 `ACCESS_TOKEN_EXPIRE_SECONDS`（默认 7200 = 2 小时）。

### Q8：中文文件名乱码

**原因：** 浏览器编码问题。

**解决：** 后端 `Content-Disposition` header 已使用 RFC 5987 编码（`filename*=UTF-8''...`），确认客户端支持。

以下脚本可以一次性测试所有 API 端点。将 `<TOKEN>` 替换为实际 JWT 值。

```bash
#!/bin/bash
BASE="http://localhost:8080/api/v1"

echo "=== 1. 注册 ==="
curl -s -X POST "$BASE/auth/register" \
  -H "Content-Type: application/json" \
  -d '{"username":"testuser","email":"test@cs.university.edu.cn","password":"Test@1234","confirm_password":"Test@1234"}' | jq

echo -e "\n=== 2. 登录 ==="
LOGIN=$(curl -s -X POST "$BASE/auth/login" \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"Admin@123456"}')
echo $LOGIN | jq

TOKEN=$(echo $LOGIN | jq -r '.data.access_token')
echo "TOKEN=$TOKEN"

echo -e "\n=== 3. 获取用户信息 ==="
curl -s "$BASE/auth/me" -H "Authorization: Bearer $TOKEN" | jq

echo -e "\n=== 4. 上传文件 ==="
echo "测试文件内容" > /tmp/test.txt
curl -s -X POST "$BASE/files/upload" \
  -H "Authorization: Bearer $TOKEN" \
  -F "file=@/tmp/test.txt" \
  -F "tags=测试,文档" \
  -F "description=测试上传" | jq
rm /tmp/test.txt

echo -e "\n=== 5. 文件列表 ==="
curl -s "$BASE/files?page=1&page_size=5" -H "Authorization: Bearer $TOKEN" | jq

echo -e "\n=== 6. 发起 SSE 对话 ==="
curl -N -X POST "$BASE/knowledge-base/chat" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -H "Accept: text/event-stream" \
  -d '{"conversation_id":null,"query":"你好，请介绍一下边缘计算","file_ids":[],"model":"gpt-4o"}'
# -N 禁用 curl 缓冲，实时看到 SSE 流

echo -e "\n=== 7. 对话列表 ==="
curl -s "$BASE/knowledge-base/conversations?page=1" -H "Authorization: Bearer $TOKEN" | jq

echo -e "\n=== 8. 看板统计（管理员） ==="
curl -s "$BASE/dashboard/stats?start_date=2026-06-01&end_date=2026-06-09" \
  -H "Authorization: Bearer $TOKEN" | jq

echo -e "\n=== 9. 看板时序 ==="
curl -s "$BASE/dashboard/stats/timeline?start_date=2026-06-01&end_date=2026-06-09&granularity=daily" \
  -H "Authorization: Bearer $TOKEN" | jq

echo -e "\n=== 10. 健康检查 ==="
curl -s http://localhost:8080/health | jq
```
