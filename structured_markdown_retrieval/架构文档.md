## 核心业务路由层：多模态文件智能鉴别引擎
来源文件：架构文档.md
文档名称：架构文档
章节标题：核心业务路由层：多模态文件智能鉴别引擎
检索关键词：架构文档 架构文档.md 核心业务路由层：多模态文件智能鉴别引擎


无论是移动端还是 Web 端，所有上传的文件和用户指令都会首先进入BFF（Backend For Frontend）业务网关层。网关层是实现您所述 3 条核心逻辑的“中央大脑”，其处理链路如下：

1. 文件格式与阈值校验网关
格式鉴权： 严格拦截校验，仅放行 MARKDOWN, XLS, PDF, PROPERTIES, TXT, VTT, MDX, CSV, XLSX, DOCX, HTML, HTM, MD 格式。
并发与容量控制： 采用令牌桶算法或队列机制，限制单批次最大处理量为 5 个文件，单文件体积上限为 15 MB。超出限制的请求直接在网关层驳回，避免穿透至底层消耗算力。

2. 动态向量距离探测机制 (Vector Distance Checking)
文件通过基础校验后，BFF 网关会提取文件摘要或首部特征文本，调用底层的向量化模型（例如高维度的 Qwen 4B Embedding 模型）将其转化为特征向量，并在向量数据库中进行相似度检索（Cosine Similarity）。
场景 A（未收录入库）：向量距离相远（低于预设相似度阈值）
判定文件为新知识。BFF 网关自动触发 Dify 的“自动化清洗入库工作流”。该工作流负责文本提取、大块分段（Chunking）、并发向量化，并最终写入向量数据库与关联元数据表。
场景 B（已收录命中）：向量距离相近（高于预设相似度阈值）
判定文件已存在于知识库。BFF 网关会拦截入库操作以避免数据冗余，将当前会话的上下文锚定至该目标文件，并向用户返回“文件已识别”的状态。此时系统进入“待命（Standby）”模式，等待用户输入具体需求后，利用 RAG 技术在目标文件中进行精准片段寻回。

3. 智能问答与 RAG 检索链路
当用户发起自然语言询问时，网关将请求连同历史会话 ID 转发至 Dify 的“知识检索与生成工作流”。系统执行混合检索（关键词 + 语义向量检索），召回高相关性切片，交由大语言模型进行逻辑加工（总结摘要、提炼关键点或定位原文件），并通过 SSE（Server-Sent Events）流式响应给终端。

## 移动端架构设计 (微信 Clawbot 接入通道)

移动端架构主打“即时通讯化”的碎片式交互体验，核心解决跨平台通道的消息转换与异步处理。

1. 交互通道层 (WeChat Client)
用户无需安装独立 App，直接在微信对话框中发送文本指令或拖拽转发支持的 13 种文件类型。微信原生提供了绝佳的用户触达便利性。

2. 消息适配网关 (Clawbot Webhook)
接入 Clawbot 作为微信通道的代理机器人。
负责将微信的 XML/JSON 消息体系解析为系统内部标准的 RESTful 数据结构。
处理微信通道的异步回调与消息重试机制。对于文件入库这种耗时操作，Clawbot 需先回复“文件处理中”的 ACK 确认，待后端入库完成后再主动推送结果消息。

3. 移动端专属会话状态机
由于微信是弱状态的 IM 通道，BFF 侧需要利用本地数据库（如 PostgreSQL）维护用户的 Session Context。当触发“场景 B（文件已存在）”时，状态机将当前用户标记为“指定文件检索模式”，确保其下一句回复能够精准路由到对应文件的 RAG 检索槽位中。

## Web 端架构设计 (CloudRAG Svelte 页面)
来源文件：架构文档.md
文档名称：架构文档
章节标题：Web 端架构设计 (CloudRAG Svelte 页面)
检索关键词：架构文档 架构文档.md Web 端架构设计 (CloudRAG Svelte 页面) Web 端架构设计 CloudRAG Svelte 页面 Web 端设计 (CloudRAG Svelte 页面) Web 端架构 (CloudRAG Svelte 页面) Web 端架构方案 Web 端架构部分


Web 端架构主打“沉浸式”的知识库管理与深度对话体验，利用现代前端框架的优势处理复杂的并发任务和数据可视化。

1. 交互通道层 (Web UI)
基于现代化前端框架构建的响应式界面，分为“文件管理看板”与“对话交互专区”。
拖拽与批量上传组件： 前端在用户选择文件阶段即执行第一层格式与大小（15MB）校验，提供实时的多线程进度条（Progress Bar）展示。

2. Web BFF 代理与 SSE 通信链路
长连接保持： 对于“场景 A”的自动处理工作流，Web 端通过轮询或 WebSocket 实时接收后端的入库进度（如：解析中 $\rightarrow$ 向量化中 $\rightarrow$ 入库成功）。
流式打字机效果： 对于“场景 C”的知识问答，前端严格通过 SSE 协议接收 Dify 工作流吐出的文本流，实现丝滑的动态打字机渲染，提供极致的 LUI 体验。

3. 沉浸式检索反馈
相比于移动端的纯文本回复，Web 端在返回加工后的总结概要时，可以利用宽屏优势，在侧边栏高亮展示通过 RAG 查找到的“原文件精确片段”及引用来源（Citations），增强大模型回答的可信度与溯源能力。
