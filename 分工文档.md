# CloudRAG-Hub（Excalibur）模块分工文档

> 版本：1.0.0 | 最后更新：2026-06-18

本项目划分为 **7 个功能模块**。每个模块的内容高度内聚——模块内的代码文件、配置和数据流向彼此直接关联。

## 模块一：项目总览与部署架构

### 负责内容
项目全局概念、系统架构设计、Docker 一键部署、环境配置、运维命令。

### 1.1 项目定位

CloudRAG-Hub（代号 Excalibur / EX咖喱棒）是一个基于 Dify Chatflow 的个人知识库 RAG 检索助手。用户通过 Web 端或微信端上传文件、提问，系统通过 Dify AI 平台从知识库中检索相关内容，交由大模型生成回答，并以流式输出的方式实时呈现。

**核心能力**：
- **RAG 知识库问答**：AI 基于用户私有知识库回答问题，逐 Token 流式输出（打字机效果）
- **文件管理**：Web 端支持文件上传/浏览/删除/下载，微信端支持文件上传和回传
- **多通道接入**：统一后端同时服务 Web 前端和微信通道
- **智能文档清洗**：定时扫描本地文件库，使用 DeepSeek 大模型将 PDF/DOCX 清洗为结构化 Markdown
- **管理员看板**：对话量、Token 消耗、延迟分布、用户排行统计
- **双管线同步**：`direct`（原文件直传）和 `deepseek`（AI 清洗后上传）两种策略

**技术栈**：

| 层级 | 技术 |
|------|------|
| 后端框架 | FastAPI (Python 3.12) + uvicorn |
| 前端框架 | Svelte 5 + Vite 8 + TailwindCSS 4 + TypeScript |
| 数据库 | PostgreSQL 15 (asyncpg 异步驱动) |
| AI 引擎 | Dify 1.14.2 (Chatflow + 知识库) |
| 大语言模型 | DeepSeek-V4 (对话 + 文档清洗) |
| Embedding | 阿里百炼 multimodal-embedding-v1 + qwen3-rerank |
| 容器化 | Docker Compose（4 容器） |
| 反向代理 | Nginx 1.27 (前端容器内置) |
| 认证 | JWT (python-jose + bcrypt) |
| 图表 | Chart.js 4 |

### 1.2 系统架构

**容器拓扑**：

```
                    ┌──────────────────────┐
                    │   cloudrag-frontend   │
                    │   Nginx :80 → :8081   │
                    └──────────┬───────────┘
                               │ proxy /api/* → backend:8080
                    ┌──────────▼───────────┐
                    │   cloudrag-backend    │
                    │   FastAPI :8080       │
                    └──┬────────┬──────────┘
                       │        │
              ┌────────▼──┐  ┌──▼──────────┐
              │ postgres  │  │ Dify (外部)  │
              │ :5433     │  │ docker_default│
              └───────────┘  │ :5001        │
                             └──────┬───────┘
                                    │
                    ┌───────────────▼───────┐
                    │  cloudrag-knowledge-   │
                    │  sync (定时扫描)       │
                    │  knowledge/ → Dify     │
                    └───────────────────────┘
```

- **cloudrag-net**（bridge）：frontend + backend + postgres + knowledge-sync 的内部网络
- **docker_default**（external）：接入 Dify 的 Docker 网络，backend 和 knowledge-sync 通过 `api:5001` 访问 Dify

**数据流全景**：

```
用户提问/上传
    │
    ▼
┌──────────────┐     ┌─────────────┐     ┌──────────────┐
│  Web 端/微信  │────▶│ 后端 BFF 层  │────▶│  Dify 工作流  │
│  (Svelte)    │     │  (FastAPI)  │     │  (Chatflow)  │
└──────────────┘     └──────┬──────┘     └──────┬───────┘
                            │                    │
                            │              ┌─────▼──────┐
                            │              │ 知识库检索   │
                            │              │ (Dataset)  │
                            │              └─────┬──────┘
                            │                    │
                            │              ┌─────▼──────┐
                            │              │ LLM 生成   │
                            │              └─────┬──────┘
                            │                    │
                            │◀─── SSE 流式透传 ───┘
                            │
              ┌─────────────▼─────────────┐
              │  sync_script.py (定时)     │
              │  knowledge/ → DeepSeek清洗 │
              │  → Dify Dataset 上传      │
              └───────────────────────────┘
```

1. 用户通过 Web/微信发起对话或上传文件
2. 后端 BFF 层鉴权后转发请求到 Dify Chatflow
3. Dify 从知识库（Dataset）检索相关文档片段
4. LLM 基于检索结果生成回答
5. 后端逐行读取 Dify 的 SSE 流，过滤处理后 SSE 透传给前端
6. 前端逐 Token 渲染，实现打字机效果
7. 同步脚本定时将 `knowledge/` 中的新文件清洗并推送到 Dify 知识库

### 1.3 Docker 容器编排

- `backend/Dockerfile`：多阶段 Python 构建，非 root 用户运行，python 健康检查

**网络**：
- `cloudrag-net`（bridge）：项目内服务间通信
- `docker_default`（external）：接入 Dify 的 Docker 网络

## 模块二：BFF 网关服务

### 负责内容
后端 FastAPI 应用的核心层：应用入口、配置管理、数据库连接、JWT 认证中间件、审计日志中间件、通用响应模型、认证/文件/管线路由。

### 2.1 应用入口（main.py）

`backend/src/main.py` — FastAPI 应用工厂，是整个后端的启动入口。

- **lifespan 管理**：启动时创建上传目录、初始化数据库 Schema、启动通道管理器；关闭时释放数据库连接池
- **CORS**：允许 `localhost:5173`、`localhost:3000`、`localhost:8081`
- **中间件**：`AuditLoggingMiddleware` 记录每次请求的耗时、状态码、IP
- **路由注册**：遍历 `routes/__init__.py` 中汇总的 7 个路由模块自动挂载
- **WebSocket**：`/ws/pipeline-status` — 客户端传入 `task_id` 后每秒轮询任务状态，变化时推送更新

### 2.2 配置管理（config.py）

`backend/src/config.py` — Pydantic Settings 类，从 `.env` 读取所有配置：

| 配置项 | 说明 |
|--------|------|
| `APP_NAME` / `APP_VERSION` | 服务标识 |
| `JWT_SECRET_KEY` / `JWT_ALGORITHM` | JWT 签名密钥和算法 (HS256) |
| `ACCESS_TOKEN_EXPIRE_SECONDS` | 访问令牌有效期（默认 7200s） |
| `REFRESH_TOKEN_EXPIRE_SECONDS` | 刷新令牌有效期（默认 604800s） |
| `POSTGRES_*` | 数据库连接参数 |
| `DIFY_BASE_URL` / `DIFY_API_KEY` | Dify API 地址和密钥 |
| `WEIXIN_BASE_URL` / `WEIXIN_TOKEN` | 微信 iLink 配置 |
| `MAX_UPLOAD_SIZE_MB` | 上传大小限制（默认 15MB） |

使用 `@lru_cache` 缓存 `get_settings()` 单例。

### 2.3 数据库连接（database.py）

`backend/src/database.py` — 基于 asyncpg 的异步连接池管理。

- `get_pool()`：懒加载创建连接池（从 Settings 读取 PostgreSQL 参数）
- `close_pool()`：关闭连接池
- `ensure_architecture_schema()`：占位函数（实际 Schema 由 `init.sql` 在容器启动时初始化）

### 2.4 JWT 认证中间件（middleware/auth.py）

`backend/src/middleware/auth.py` — 无状态 JWT 认证体系。

- `create_access_token()` / `create_refresh_token()`：签发 JWT，Payload 包含 `sub`（用户 UUID）、`username`、`role`、`iat`、`exp`
- `decode_token()`：验证并解码 JWT，失败返回 401
- `get_current_user()`：FastAPI 依赖注入，从 `Authorization: Bearer` 头提取 token 并解码，返回用户载荷。`auto_error=False` 表示 token 缺失时不强制报错，由各路由自行判断是否要求认证

**安全设计**：
- 密码使用 PostgreSQL `pgcrypto` 扩展的 `crypt()` 函数进行 bcrypt 哈希和验证，密码明文不经过应用层
- 密码强度要求 8-64 字符，含大小写、数字、特殊字符中的至少三种

### 2.5 审计日志中间件（middleware/logging.py）

`backend/src/middleware/logging.py` — 请求级审计追踪。

- `AuditLoggingMiddleware`：为每个请求生成 UUID（`X-Request-ID`），记录耗时（`X-Latency-Ms`）并附加响应头
- `log_request_to_db()`：将请求信息（用户、方法、路径、状态码、耗时、IP、UA）写入 `operation_logs` 表，供模块七的 Dashboard 统计使用

### 2.6 通用响应模型（models/）

`backend/src/models/__init__.py` — 统一的数据传输对象：

- **Envelope**：标准化响应信封 `{code, message, data, meta}`，提供 `success()` 和 `error()` 工厂方法。所有非 SSE 接口均使用此格式
- **PaginatedData**：分页数据结构 `{items, total, page, page_size, total_pages}`
- **UserBrief**：用户基本信息（id, username, email, avatar_url, role）
- **FileInfo**：文件元数据（original_name, mime_type, size_bytes, processing_status, created_at）
- **ConversationBrief**：对话列表摘要（title, model, message_count, last_message_preview, total_tokens）
- **MessageItem**：消息记录（role, content, references, usage, latency_ms）

### 2.7 认证路由（routes/auth.py）

`/api/v1/auth` 前缀：

| 端点 | 说明 |
|------|------|
| `POST /login` | 用户名+密码登录，PostgreSQL 端 bcrypt 验证，返回 access_token + refresh_token + 用户信息 |
| `POST /register` | 注册，校验密码一致性、用户名/邮箱唯一性，bcrypt 哈希存储 |
| `POST /refresh` | 刷新令牌，验证 refresh_token 后发放新 token 对 |
| `GET /me` | 获取当前用户信息，包含配额使用量 |

### 2.8 文件管理路由（routes/files.py）

`/api/v1/files` 前缀：

| 端点 | 说明 |
|------|------|
| `GET /` | 分页查询当前用户文件列表，支持按 `folder` 过滤，查询 `file_metadata` 表，返回 `size_human` 可读格式 |
| `DELETE /{file_id}` | 软删除文件（设置 `deleted_at`），同时归还用户存储配额（`quota_used_bytes -= file.size_bytes`） |

文件查询使用 `file_metadata_active` 视图（过滤 `deleted_at IS NULL`），用户只能操作自己上传的文件。

### 2.9 管线路由（routes/pipeline.py）

`/api/v1/pipeline` 前缀。当前为占位实现：`GET /status/{task_id}` 返回固定值 `"unknown"`，管线任务的实际进度通过 WebSocket（`/ws/pipeline-status`）推送。

### 2.10 辅助服务

**文件匹配服务（services/file_matcher.py）**：使用 `rapidfuzz` 进行文件名模糊匹配。递归遍历知识库目录，过滤临时文件（`~$` 前缀），支持中文自然语言查询（如"打开 架构文档" → 剥离"打开"前缀 → 匹配"架构文档.docx"）。`resolve_best_match()` 先精确匹配再模糊匹配，`search_knowledge_files()` 返回 top-K 评分超过 70% 的结果。

**动态 RAG 配置（services/dynamic_rag.py）**：对话文件上传的校验常量。

**管线任务存储（services/pipeline_store.py）**：内存键值存储，供 WebSocket 端点轮询任务进度。

**Dify 文档同步（services/dify_sync.py）**：占位模块（`sync_dify_documents_to_cloudrag()` 为 `pass`），预留 Dify ↔ CloudRAG 双向同步入口。

## 模块三：前端 Web 交互界面

### 负责内容
基于 Svelte 5 的单页应用（SPA），包括组件树、SSE 流式渲染、API 客户端、状态管理、Nginx 反向代理。

### 3.1 路由与入口

`frontend/src/App.svelte` — 根组件，使用 **Hash 路由**（`#/chat`、`#/dashboard`、`#/channels`、`#/login`、`#/register`）：

| 路由 | 页面 | 权限 |
|------|------|------|
| `#/login` | 登录页 | 公开 |
| `#/register` | 注册页 | 公开 |
| `#/chat` | 对话页（默认） | 需登录 |
| `#/dashboard` | 数据看板 | 管理员 |
| `#/channels` | 通道管理 | 需登录 |

未登录用户（除注册页）自动重定向到登录页。使用 `$effect` 监听 `isLoggedIn` store 和 `hashchange` 事件。

### 3.2 组件树

```
App.svelte
├── Toast.svelte                          # 全局通知（success/error/info，4s 自动消失）
├── Login.svelte                          # 登录/注册（未登录时）
└── AppLayout.svelte                      # 主布局（已登录时）
    ├── Sidebar.svelte                    # 侧边栏：对话历史列表（最多50条）、右键删除、用户信息、登出
    ├── MobileNav.svelte                  # 底部移动导航
    └── (路由内容区)
        ├── ChatView.svelte               # 对话界面（核心）
        ├── Dashboard.svelte              # 数据看板
        ├── ChannelManager.svelte         # 通道管理
        └── FileManager.svelte            # 文件管理
```

### 3.3 ChatView.svelte — 知识库对话界面

`frontend/src/components/ChatView.svelte`（501 行），前端最核心组件。

**对话功能**：
- 渲染历史消息（用户/助手），Markdown 支持（粗体 `**text**`、二级/三级标题、有序/无序列表）
- 引用展示：可折叠卡片，显示文件名、页码、评分百分比、内容摘要
- Token 用量和延迟脚注
- 模型选择：`deepseek-v4-flash`（硬编码）
- 新建/打开历史/停止生成

**SSE 流式渲染**：
- 通过 `sse.ts` 的 `createChatStream()` 发起请求
- `createChatStream()` 回调实时更新 `streamingState`
- `streamingState.content` 逐 Token 追加，闪烁光标动画（`cursor-blink` CSS keyframe）
- 流结束后将累积文本整体渲染为 Markdown

**文件上传**：
- 点击 Paperclip 图标或拖拽，校验文件类型（白名单 13 种）和大小（15MB）
- 单批最多 5 个文件，每天最多 20 次（localStorage 计数）
- 调用 `kbApi.uploadChatFiles()` → 后端 `POST /chat/upload` → Dify `/files/upload`
- 上传进度条 + 同步状态标签（"同步中"/"已归档"）

### 3.4 FileManager.svelte — 文件管理

`frontend/src/components/FileManager.svelte`（382 行）。

**功能特性**：
- 分页网格展示，每行：图标（按 MIME 类型）、文件名、大小、时间、状态标签（就绪/处理中/等待中/失败）、文件夹标签
- 搜索过滤：关键词搜索文件名/文件夹，按状态筛选
- 批量上传：拖拽 + 文件选择器 + 文件夹上传，单批 5 个，单文件 15MB
- 分段设置面板：可展开配置分段标识符（`\n\n`/`\n`/`。/`. `等等`）、最大长度（100-2000 滑块）、重叠长度（0-200 滑块）
- 文件操作：下载（`filesApi.downloadUrl()`）、删除（需确认）
- Dify 同步：每 5 秒自动 `syncFromDify()` 保持 CloudRAG 和 Dify 文件列表一致

## 模块四：RAG 对话与 AI 引擎

### 负责内容
Dify AI 平台集成、Chatflow 工作流、SSE 流式对话、知识库检索、检索质量优化。这是项目最核心的"AI 大脑"模块。

### 4.1 Dify 工作流（EX咖喱棒.yml）

`EX咖喱棒.yml`（1344 行）— Dify 高级对话模式的 Chatflow DSL，共 22 个节点。

**工作流结构**：

```
start (用户输入)
  │
  ▼
question-classifier (问题分类器: deepseek-v4-flash)
  ├─ "内容查询" → LLM 2(查询改写器) → knowledge-retrieval → LLM(答案生成) → answer
  ├─ "文件操作" → LLM 3(关键词提取) → HTTP_Search → 代码执行4 → assigner → if-else
  │    ├─ count=0 → LLM 4 → answer (未找到)
  │    ├─ count=1 → 代码执行 → HTTP_OPEN → LLM 5 → answer (找到)
  │    └─ count>1 → LLM 6 → answer (多选列表)
  ├─ "按序号打开" → 代码执行3 → HTTP_OPEN → LLM 6 → answer
  └─ "文件上传" → HTTP 请求4 → LLM 8 → answer
```

**核心节点配置**：

| 节点 | 模型/配置 | 功能 |
|------|-----------|------|
| question-classifier | deepseek-v4-flash | 将输入分为 4 类：内容查询/文件操作/按序号打开/文件上传 |
| LLM 2（查询改写器） | deepseek-v4-flash, t=0.1 | 改写问题以提高召回率，补充同义词和文件名变体 |
| knowledge-retrieval | multimodal-embedding-v1 + qwen3-rerank | 混合搜索（向量 0.5/关键词 0.5），top_k=20，重排序 |
| LLM（答案生成） | deepseek-v4-flash, max_tokens=4096, t=0.3 | 基于检索上下文生成回答 |
| HTTP_Search | `http://host.docker.internal:8080/search` | 调用 agent_server 搜索文件 |
| HTTP_OPEN | `http://host.docker.internal:8080/open` | 调用 agent_server 打开文件 |

工作流通过 HTTP 节点调用 agent_server（模块五），实现了 Dify 与本地文件库的联动——用户不仅可以通过 Dify 聊天，还能直接操作本地文件。

### 4.2 Dify 网关服务（services/dify.py）

`backend/src/services/dify.py`（598 行）— 后端与 Dify 交互的核心服务层。

**DifyGateway 类**：

- `upload_file()`：调用 Dify `/files/upload` API，将用户上传的对话附件推送到 Dify 文件存储，返回文件 ID
- `stream_chat_messages()`：构建 Dify `/chat-messages` 请求体，包含用户输入、对话 ID、文件列表等参数

**chat_stream() 生成器** — SSE 事件生成全流程：

1. 在数据库中新建或续写对话记录（写入 `conversations` 和 `messages` 表）
2. 发送 `meta` 事件（conversation_id、model、created_at、user_id）
3. 发送空 `references` 事件（占位，实际引用由后续节点填充）
4. 向 Dify 发起流式 POST 请求（`response_mode: "streaming"`），逐行读取 SSE 响应
5. **ThinkBlockFilter**：过滤 DeepSeek 推理过程中 `<｜end▁of▁thinking｜>` 标签内的思考内容，只向用户展示最终回答
6. 从 Dify 返回的 `knowledge-retrieval` 节点中提取参考文献
7. **相关性评分**（`_filter_visible_references()`）：
   - 优先展示文件名出现在用户问题中的 chunk
   - 计算 chunk 内容与问题的词汇匹配得分
   - 按得分降序排列并去重
8. 发送 `message` 事件（逐 Token 增量）、`references` 事件（过滤去重后的检索结果）
9. 发送 `done` 事件（message_id、usage{tokens}、latency_ms、model、finished_at）

### 4.3 对话路由（routes/chat.py）

`backend/src/routes/chat.py`（247 行），`/api/v1/knowledge-base` 前缀：

| 端点 | 说明 |
|------|------|
| `POST /chat/upload` | 上传对话附件到 Dify。限制：单次最多 5 个文件，单个最大 15MB，校验扩展名白名单，当日最多 20 次。调用 `DifyGateway.upload_file()` |
| `POST /chat` | **核心对话端点**。接收 `ChatRequest`，返回 SSE 流（`text/event-stream`）。调用 `chat_stream()` 生成 6 种 SSE 事件 |
| `GET /conversations` | 分页查询当前用户的对话列表，含最后消息摘要、token 总数 |
| `GET /conversations/{id}` | 查询单条对话完整详情，包含所有的 messages 数组 |
| `DELETE /conversations/{id}` | 删除对话及其关联的所有消息记录 |

### 4.4 SSE 事件模型（models/chat.py）

`backend/src/models/chat.py` — 严格对应 `api-contract.md` 第 5 节定义的 6 种 SSE 事件：

| 模型 | 字段 | 触发时机 |
|------|------|----------|
| `SSEMetaEvent` | conversation_id, model, created_at, user_id | 流开始，第一个事件 |
| `SSEReferencesEvent` | chunks[], total_retrieved | 检索完成后，回答开始前 |
| `SSEMessageToken` | token(string), index(int) | 逐 Token 增量推送，实现打字机效果 |
| `SSEToolCallEvent` | tool_name, status, input, output, duration_ms | Agent 调用外部工具时 |
| `SSEErrorEvent` | code, message, conversation_id | 发生错误时 |
| `SSEDoneEvent` | conversation_id, message_id, usage, latency_ms, model, finished_at | 对话完成 |

`ChatRequest` 请求体包含：conversation_id(可null)、query(1-4000字符)、file_ids、model、retrieval_config(top_k/score_threshold/rerank_enabled)、system_prompt_override。

### 4.5 知识库与检索优化

**知识库架构**：

- **knowledge/**：本地文件库物理存储，Web/微信上传的文件落在此处，同步脚本以此为源
- **Dify Dataset**：Dify 的知识库，同步脚本将 `knowledge/` 中的文件清洗后上传到此，Dify 工作流的 `knowledge-retrieval` 节点从此检索
- **structured_markdown/**：清洗后的结构化 Markdown 中间产物（由同步脚本生成）

**四层检索质量保障**：

1. **DeepSeek 智能清洗**：通过约 1100 字的系统提示词，保留技术细节（实验数据、配置参数、代码块），消除打印噪声（页眉页脚、页码、水印），修复表格格式，实施"语义锚点拼接"——将子序号升级为 `##` 标题并用父级标题前缀拼接（如 `## 一、引言段 - 1. 交代背景`）
2. **内容保真度校验**：清洗后与原文做 `SequenceMatcher` 相似度对比，低于 95% 则回退到原文保留模式
3. **Dify 分段策略**：deepseek 管线使用 `\n## ` 作为分段分隔符（而非默认的 `\n\n`），确保每个 Chunk 是一个独立语义单元，每段最大 1400 tokens
4. **重排序优化**：Dify 工作流启用 qwen3-rerank 重排序模型，对 top_k=20 的候选 chunk 进行二次精排

## 模块五：文件同步与数据处理管线

### 负责内容
本地文件库 → DeepSeek 清洗 → Dify Dataset 的完整数据管线，以及本地文件 HTTP 服务的实现。

### 5.1 文件同步脚本（sync_script.py）

`sync_script.py`（1230 行）— 独立于 CloudRAG 后端的知识库同步脚本，可命令行运行或容器化运行。

**双管线设计**：

| 管线 | 参数 | 流程 | 适用 |
|------|------|------|------|
| `direct` | `--pipeline direct` | 原文件直接上传到 Dify Dataset，Dify 自动分段 | 纯文本/md 文件 |
| `deepseek` | `--pipeline deepseek` | 提取文本 → DeepSeek API 清洗 → 结构化 Markdown → 上传 | PDF/DOCX 等排版混乱的文档 |

**DeepSeek 清洗管线五步流程**：

**步骤 1 — 文本提取**：
- DOCX：`python-docx`，支持段落和表格混合提取，表格以 Tab 分隔（`\t`）
- PDF：`PyPDF2`，逐页提取文本
- MD/TXT：直接 UTF-8 读取
- 输出统一 `sanitize_text_for_json()` 确保编码兼容

**步骤 2 — DeepSeek 结构化清洗**：
- 使用 OpenAI 兼容 `/chat/completions` 接口，可替换为任意兼容端点（DeepSeek、阿里百炼、通义千问等）
- 系统提示词（`DEEPSEEK_SYSTEM_PROMPT`）定义详尽规则：
  - 实验报告类：严格保留实验目的、步骤、数据、日志、结论
  - 实验模板类：保留引导文字和占位符（`[请在此处填写]`）
  - 架构设计类：侧重组件关系、协议流程、配置项
  - 章节锚点拼接公式：`## [父级大板块标题] - [子序号] [子标题]`
  - 表格修复：乱序线条重排为标准 Markdown 表格
  - 代码块强制围栏包裹
- 超长文本分段发送，每段 ≤ 18000 字符

**步骤 3 — 内容保真度校验**：
- `markdown_body_similarity()`：`SequenceMatcher` 计算清洗后 Markdown 与原文的正文相似度
- 低于阈值（默认 95%）则回退到 `preserved_source_markdown()`：保留原文每一行，Tab 格式自动转 Markdown 表格
- `ensure_markdown_body_fidelity()`：门控函数，自动选择清洗结果或保真回退

**步骤 4 — 检索增强处理**：
- `promote_structural_headings()`：中文序号标题（"一、xxx"、"第X章"）自动升级为 `##` 二级标题
- `enhance_markdown_for_retrieval()`：综合调用标题升级和元数据剥离，输出纯净可检索的 Markdown

**步骤 5 — 上传到 Dify**：
- deepseek 管线使用自定义分段：`\n## ` 分隔，每段最大 1400 tokens
- direct 管线使用 Dify 自动分段模式（`mode: "automatic"`）

**同步账本（Ledger）机制**：

- **存储**：`sync_ledger.json`（docker-compose 将其挂载到容器的 `/app/state/` 目录）
- **数据结构**：`{files: {文件路径: {sha256, size, status, document_id, pipeline, ...}}, whitelist: {}}`
- **增量检测**：比较文件 SHA256 和大小，仅上传发生变化的文件，避免重复上传
- **远程对账**：`refresh_ledger_whitelist()` 拉取 Dify Dataset 已有文档列表，与本地文件匹配，自动同步状态
- **状态流转**：`synced`（已同步）→ 文件变化 → `pending`（待上传）→ 上传成功 → `synced` 或 `failed`
- **Pipeline 版本追踪**：deepseek 管线记录 `pipeline_version`，升级后自动触发全量重建

**支持的文件格式**：`.csv`、`.doc`、`.docx`、`.htm`、`.html`、`.markdown`、`.md`、`.mdx`、`.pdf`、`.properties`、`.txt`、`.vtt`、`.xls`、`.xlsx`（共 14 种）

### 5.4 文件模糊匹配（services/file_matcher.py）

`backend/src/services/file_matcher.py`（151 行）— 跨模块共享的文件搜索工具。

- `iter_knowledge_files()`：递归遍历知识库目录，跳过临时文件（`~$` 前缀）

- `resolve_best_match()`：三级匹配策略：精确大小写匹配 → 大小写不敏感 → 无扩展名匹配 → 模糊匹配

**ChannelManager** — 通道生命周期管理器：

- **双通道管理**：`weixin`（微信 iLink）和 `webhook`（通用 Webhook）
- **启动流程**：`startup()` → 加载凭据文件 → 若凭据有效则自动启动微信轮询 → 否则进入待连接状态
- **连接流程**：`connect()` → `create_qr_login()` 获取二维码 → 生成 data URI 图片 → 轮询确认 → 保存凭据 → 启动轮询
- **轮询循环**：`start_weixin_polling()` 创建后台异步任务，固定间隔调用 `poll_weixin_once()`
- **消息去重**：基于消息 ID 和内容哈希的去重机制
- **消息处理**：`process_weixin_update()` → 标准化原始载荷 → 若为文件消息则下载解密 → 委托给 `handle_weixin_event()`
- **文件操作上下文**：`remember_file_choices()` / `resolve_file_choice()` 维护用户的文件选择状态（当 Dify 返回多个匹配文件时，用户可输入"打开第X个"）
- **凭据持久化**：`_save_weixin_credentials()` / `_load_weixin_credentials()` 将 bot token 保存到 JSON 文件，重启后自动恢复连接

**加密传输**：微信通道的文件使用 AES-ECB 加密：

```
发送文件: 文件内容 → AES-ECB加密 → 微信CDN → 加密参数 → 微信用户(微信客户端解密)
接收文件: 微信用户 → 微信CDN加密 → 后端下载 → AES-ECB解密 → 保存到knowledge/
```

加密函数 `_aes_encrypt()` 和 `_aes_decrypt()` 通过 `pycryptodome` 库实现。

### 6.2 通道管理路由（routes/channels.py）

`backend/src/routes/channels.py`（347 行），`/api/v1/channels` 前缀：

| 端点 | 说明 |
|------|------|
| `GET /` | 列出所有通道及状态（名称、图标、连接状态、运行状态） |
| `POST /` | 连接/断开通道（`action: "connect"` / `"disconnect"`） |
| `POST /weixin/qrlogin` | 微信扫码登录。action：`fetch` 获取二维码 → `poll` 等待扫码 → `refresh` 刷新 |
| `POST /weixin/webhook` | 处理微信入站事件 |

**核心函数 `handle_weixin_event()`** 处理三类入站事件：

| 事件类型 | 处理逻辑 |
|----------|----------|
| **open_file**（打开文件） | 根据文件名在知识库中模糊匹配 → 找到则将文件加密上传微信 CDN → 发送给用户 |
| **text**（文本消息） | 先在知识库模糊搜索匹配文件 → 有匹配返回文件选项列表（让用户选第几个）→ 无匹配则转发到 Dify 获取 AI 回答 |
| **file**（文件消息） | 从微信 CDN 下载并解密 → 保存到 `knowledge/` 目录 → 确认上传成功 → 后续由同步脚本（模块五）推送到 Dify Dataset |

### 6.3 ClawBot 兼容路由（routes/clawbot.py）

`backend/src/routes/clawbot.py`（160 行），`/api/v1/clawbot` 前缀。为 ClawBot/iLink 格式提供兼容端点：

| 端点 | 说明 |
|------|------|
| `GET /status` | 健康检查 |
| `GET /weixin-state` | 返回微信通道状态（connected、running、polling、cursor、has_token） |
| `POST /webhook` | 接收 ClawBot 格式 Webhook，`_process_clawbot_payload()` 标准化并去重后调用 `handle_weixin_event` |
| `POST /sendmessage` | 同 webhook 逻辑 |
| `GET\|POST /getupdates` | 轮询端点；空消息返回空列表，有消息则处理 |
| `GET\|POST /sendtyping` | 占位端点 |
| `GET\|POST /getconfig` | 返回 accepted 状态，含长轮询超时提示 |

`_process_clawbot_payload()` 将 ClawBot 的原始载荷（各种字段名变体）统一转换为标准化格式，再进行消息去重，最后交给 `handle_weixin_event` 统一处理。

## 模块七：数据看板与分析引擎

### 负责内容
PostgreSQL 数据库 Schema 设计、管理员数据分析看板、审计日志追踪、统计视图。

### 7.1 数据库 Schema 设计（init.sql）

`backend/init.sql`（246 行）— 容器首次启动时由 PostgreSQL 自动执行。

**七张核心表**：

| 表名 | 说明 | 核心字段 |
|------|------|----------|
| `users` | 用户表 | id(UUID), username(唯一), email(唯一), password_hash(bcrypt), role(admin/user), quota_used_bytes, quota_total_bytes(1GB), is_active, last_login_at |
| `file_metadata` | 文件元数据 | original_name, stored_name, mime_type, size_bytes, tags(TEXT[]), folder, processing_status(pending/processing/ready/failed), duplicate_of(自引用FK), dify_document_id, uploaded_by(FK users), deleted_at(软删除) |
| `file_vectors` | 文件级向量 | file_id(FK), embedding(JSONB, 2560维), model(qwen3-embedding:4b) |
| `mobile_session_context` | 移动端会话 | external_user_id(PK), mode(idle/open_file...), target_file_id(FK), updated_at |
| `conversations` | 对话表 | title, model, user_id(FK), created_at, updated_at |
| `messages` | 消息表 | conversation_id(FK), role(user/assistant/system/tool), content, prompt_tokens, completion_tokens, total_tokens, latency_ms, references_json(JSONB), tool_calls_json(JSONB) |
| `operation_logs` | 操作日志 | user_id(FK), request_id, method, path, status_code, latency_ms, ip_address, user_agent, token_used |

**两个统计视图**：

- **v_user_stats**：用户用量概览（对话数、文件数、Token 消耗、消息数、最后登录），从四张表联合聚合
- **v_daily_stats**：每日统计（对话数、总请求数、AI 请求数、平均延迟、错误数、活跃用户数），从 `operation_logs` 按日期聚合

**种子数据**：

| 用户名 | 密码 | 角色 | 配额 |
|--------|------|------|------|
| `admin` | `Admin@123456` | admin | 10GB |
| `demo_user` | `Demo@123456` | user | 1GB |

密码通过 `pgcrypto` 扩展的 `crypt()` 函数以 bcrypt (blowfish, 10 rounds) 哈希存储。

**索引策略**：用户查询索引（username/email/role）、文件多维索引（uploaded_by/status/mime_type/folder，GIN 索引支持 tags 数组）、对话时间排序、消息三层索引（conversation_id/created_at/role）、日志多维统计索引（user_id/时间/方法路径/状态码/延迟/Token 标记）、`updated_at` 自动更新触发器。

### 7.2 管理看板路由（routes/dashboard.py）

`backend/src/routes/dashboard.py`（349 行），`/api/v1/dashboard` 前缀。**所有端点仅管理员可访问**，通过 `_require_admin()` 门控。

| 端点 | 说明 | SQL 技术 |
|------|------|----------|
| `GET /overview` | 概览数据（对话总数） | 简单 COUNT |
| `GET /stats` | 综合统计：对话量、消息量、Token 消耗（输入/输出/总计）、预估费用（`input×0.0000005 + output×0.0000015`）、延迟分布（avg/p50/p95/p99/min/max，使用 PostgreSQL `percentile_cont`）、文件上传量、存储空间、活跃用户数、模型用量分布 | 多表 JOIN + 聚合 + 百分位函数 |
| `GET /stats/timeline` | 时间序列：按天聚合，使用 `generate_series` 补全无数据日期，确保折线图连续 | `generate_series` + LEFT JOIN |
| `GET /stats/users` | 用户排名：按 Token 消耗/对话数/消息数排序，含最后活跃时间 | GROUP BY + ORDER BY + LIMIT |

**费用估算公式**：输入 Token `$0.0005/M` + 输出 Token `$0.0015/M`（基于 DeepSeek 定价）。

**延迟百分位计算**：使用 PostgreSQL `percentile_cont()` 聚合函数，准确计算 p50/p95/p99。

**日期边界处理**：`_date_bounds()` 确保 `start_date ≤ end_date` 且 `end_date` 包含当天（`+1 day` 半开区间）。

### 7.3 审计日志体系

审计日志从请求进入（模块二的 `AuditLoggingMiddleware`）到持久化（`log_request_to_db()`）再到统计分析（dashboard 路由的 SQL 查询）形成闭环：

```
HTTP 请求进入
  │
  ▼
AuditLoggingMiddleware  →  生成 request_id  →  记录耗时  →  附加响应头
  │
  ▼
log_request_to_db()  →  INSERT INTO operation_logs
  │
  ▼
Dashboard SQL 查询 ← SELECT ... FROM operation_logs  (按日期/用户/状态码聚合)
```

`operation_logs` 表同时承载 Dashboard 的所有统计查询负载，通过 7 个复合索引优化查询性能。

### 7.4 数据库连接池

`backend/src/database.py` 使用 asyncpg 的连接池模式：
- 所有路由通过 `await get_pool()` 获取连接
- Dashboard 统计查询直接使用 `pool.fetch()` / `pool.fetchrow()` / `pool.fetchval()`
- 文件删除操作使用 `pool.acquire()` 获取连接并开启事务（软删除 + 配额归还在同一事务中）
- 应用关闭时 `close_pool()` 释放所有连接
