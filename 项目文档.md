# CloudRAG-Hub（Excalibur）项目文档

> 版本：1.0.0 | 最后更新：2026-06-18

---

## 1. 项目概述

CloudRAG-Hub（代号 Excalibur / EX咖喱棒）是一个基于 Dify Chatflow 的个人知识库 RAG 检索助手。用户通过 Web 端或微信端上传文件、提问，系统通过 Dify AI 平台从知识库中检索相关内容，交由大模型生成回答，并以流式输出的方式实时呈现。

### 1.1 核心能力

- **RAG 知识库问答**：上传文档后，AI 基于用户私有知识库回答问题，逐 Token 流式输出（打字机效果）
- **文件管理**：Web 端支持文件上传/浏览/删除/下载，微信端支持文件上传和回传
- **多通道接入**：统一后端同时服务 Web 前端和微信通道（通过 iLink / ClawBot 轮询）
- **智能文档清洗**：定时扫描本地文件库，使用 DeepSeek 大模型将 PDF/DOCX 清洗为结构化 Markdown，上传到 Dify 知识库
- **管理员看板**：对话量、Token 消耗、延迟分布、用户排行等统计图表
- **双管线同步**：支持 `direct`（原文件直传）和 `deepseek`（AI 清洗后上传）两种知识库同步策略

### 1.2 技术栈一览

| 层级 | 技术 |
|------|------|
| 后端框架 | FastAPI (Python 3.12) + uvicorn |
| 前端框架 | Svelte 5 + Vite 8 + TailwindCSS 4 + TypeScript |
| 数据库 | PostgreSQL 15 (asyncpg 异步驱动) |
| AI 引擎 | Dify 1.14.2 (Chatflow + 知识库) |
| 大语言模型 | DeepSeek-V4 (对话 + 文档清洗) |
| Embedding | 阿里百炼 multimodal-embedding-v1 + qwen3-rerank |
| 容器化 | Docker Compose（4 容器：frontend + backend + knowledge-sync + postgres） |
| 反向代理 | Nginx 1.27 (前端容器内置) |
| 认证 | JWT (python-jose + bcrypt) |
| 图表 | Chart.js 4 |

### 2.2 数据流全景

```
用户提问/上传
    │
    ▼
┌──────────────┐     ┌─────────────┐     ┌──────────────┐
│  Web 端/微信  │────▶│ 后端 BFF 层  │────▶│  Dify 工作流  │
│  (Svelte)    │     │  (FastAPI)  │     │  (Chatflow)  │
└──────────────┘     └──────┬──────┘     └──────┬───────┘
                            │                    │
                            │              ┌─────▼──────┐
                            │              │ 知识库检索   │
                            │              │ (Dataset)  │
                            │              └─────┬──────┘
                            │                    │
                            │              ┌─────▼──────┐
                            │              │ LLM 生成   │
                            │              └─────┬──────┘
                            │                    │
                            │◀─── SSE 流式透传 ───┘
                            │
              ┌─────────────▼─────────────┐
              │  sync_script.py (定时)     │
              │  knowledge/ → DeepSeek清洗 │
              │  → Dify Dataset 上传      │
              └───────────────────────────┘
```

1. 用户通过 Web/微信发起对话或上传文件
2. 后端 BFF 层鉴权后转发请求到 Dify Chatflow
3. Dify 从知识库（Dataset）检索相关文档片段
4. LLM 基于检索结果生成回答
5. 后端逐行读取 Dify 的 SSE 流，过滤处理后再 SSE 透传给前端
6. 前端逐 Token 渲染，实现打字机效果
7. 同步脚本定时将 `knowledge/` 中的新文件清洗并推送到 Dify 知识库

后端是 BFF（Backend For Frontend）层，不直接处理 RAG 逻辑，只负责鉴权、通道适配、请求转发、文件落库和流式透传。

## 5. Dify AI 工作流（EX咖喱棒.yml）

`EX咖喱棒.yml:1344` — Dify 高级对话（advanced-chat）模式的工作流 DSL

### 5.1 工作流结构

22 个节点组成的有向图：

```
start (用户输入)
  │
  ▼
question-classifier (问题分类器)
  ├─ "内容查询" ──▶ LLM 2 (查询改写器) ──▶ knowledge-retrieval ──▶ LLM (RAG 答案生成) ──▶ answer
  ├─ "文件操作" ──▶ LLM 3 (关键词提取) ──▶ HTTP_Search ──▶ 代码执行4 ──▶ assigner ──▶ if-else
  │                    ├─ count=0 ──▶ LLM 4 ──▶ answer (未找到)
  │                    ├─ count=1 ──▶ 代码执行 ──▶ HTTP_OPEN ──▶ LLM 5 ──▶ answer (找到)
  │                    └─ count>1 ──▶ LLM 6 ──▶ answer (多选列表)
  ├─ "按序号打开" ──▶ 代码执行3 ──▶ HTTP_OPEN ──▶ LLM 6 ──▶ answer
  └─ "文件上传" ──▶ HTTP 请求4 ──▶ LLM 8 ──▶ answer (上传成功)
```

### 5.2 核心节点配置

| 节点 | 模型/配置 | 说明 |
|------|-----------|------|
| question-classifier | deepseek-v4-flash | 将用户输入分为 4 类：内容查询、文件操作、按序号打开文件、文件上传 |
| LLM 2（查询改写器） | deepseek-v4-flash, temperature=0.1 | 将用户问题改写为更适合检索的查询词，补充同义词和文件名变体 |
| knowledge-retrieval | multimodal-embedding-v1 + qwen3-rerank | 混合搜索（向量权重 0.5 / 关键词权重 0.5），top_k=20，开启重排序 |
| LLM（答案生成） | deepseek-v4-flash, max_tokens=4096, temperature=0.3 | 基于检索到的上下文生成回答，指令强调只围绕用户问题回答 |
| LLM 3（关键词提取） | deepseek-v4-flash | 从用户输入提取文件关键词，结构化输出 `{"keyword": "string"}` |
| HTTP_Search | `http://host.docker.internal:8080/search` | 调用 agent_server 的文件搜索接口 |

## 6. 文件同步管线（sync_script.py）

`sync_script.py:1230` — 独立于 CloudRAG 后端的知识库同步脚本，可命令行运行或容器化运行。

### 6.1 双管线设计

| 管线 | 参数值 | 流程 | 适用场景 |
|------|--------|------|----------|
| `direct` | `--pipeline direct` | 原文件直接上传到 Dify Dataset | 纯文本/md 文件，无需清洗 |
| `deepseek` | `--pipeline deepseek` | 提取文本 → DeepSeek 清洗为结构化 Markdown → 上传 | PDF/DOCX 等排版混乱的文档 |

### 6.2 DeepSeek 清洗管线详解

1. **文本提取**：
   - DOCX：使用 `python-docx`，支持段落和表格混合提取，表格以 Tab 分隔
   - PDF：使用 PyPDF2，逐页提取文本
   - MD/TXT：直接读取 UTF-8 文本
   - 输出统一经过 `sanitize_text_for_json()` 确保 UTF-8 兼容

2. **DeepSeek 结构化清洗**：
   - 使用 OpenAI 兼容 `/chat/completions` 接口，可替换为 DeepSeek、阿里百炼等兼容端点
   - 系统提示词（`DEEPSEEK_SYSTEM_PROMPT`，约 1100 字符）定义了详尽的清洗规则：
     - **实验报告类**：严格保留实验目的、步骤、数据、命令日志、结论
     - **实验模板类**：保留引导性文字和占位符（如 `[请在此处填写]`）
     - **架构设计类**：侧重组件关系、协议流程、配置项
     - **章节锚点拼接**：将子序号升级为 `##` 标题，父级标题作为前缀拼接（如 `## 一、引言段 - 1. 交代背景`），确保每个 Chunk 包含完整语义锚点
     - **表格修复**：乱序表格线重排为标准 Markdown 表格
   - 超长文本分段发送，每段不超过 18000 字符

3. **内容保真度校验**：
   - `markdown_body_similarity()`：计算清洗后 Markdown 与原始文本的正文相似度（基于 `SequenceMatcher`）
   - 若相似度低于阈值（默认 95%），回退到 `preserved_source_markdown()`，保留原文每一行
   - `ensure_markdown_body_fidelity()`：门控函数，自动选择清洗结果或保真回退

4. **检索增强处理**：
   - `promote_structural_headings()`：将中文序号标题（如"一、xxx"、"第X章"）升级为 `##` 二级标题
   - `enhance_markdown_for_retrieval()`：调用标题升级和检索提示剥离
   - `strip_retrieval_hints()`：移除历史遗留的检索注释和元数据行

5. **上传到 Dify**：
   - deepseek 管线使用自定义分段规则（`\n## ` 分隔，每段最大 1400 tokens）
   - direct 管线使用 Dify 自动分段

### 6.3 同步账本机制（Ledger）

- **存储位置**：`SYNC_LEDGER_PATH`（默认 `sync_ledger.json`）
- **数据结构**：`{files: {文件路径: {sha256, size, status, document_id, ...}}, whitelist: {}}`
- **增量检测**：比较文件的 SHA256 和大小，仅上传发生变化的文件
- **远程对账**：`refresh_ledger_whitelist()` 获取 Dify Dataset 中已有文档列表，匹配本地文件，更新同步状态
- **状态流转**：`pending` → `synced` / `failed`

### 6.4 容器化运行

`knowledge-sync.Dockerfile` — 基于 Python 3.12-slim，安装 `requests`、`python-docx`、`PyPDF2`

`knowledge-sync-entrypoint.sh:40` — 入口脚本逻辑：

### 10.1 knowledge/ 目录

本地文件库的物理存储位置，Web 上传、微信上传、手动导入的文件都落在此处。同步脚本扫描此目录，将支持的文件（见 `SUPPORTED_EXTENSIONS`，包含 pdf/docx/md/txt/csv/xlsx 等 16 种格式）同步到 Dify Dataset。

### 10.3 structured_markdown_retrieval/ 目录

### 10.4 检索质量保障

多层保障确保 RAG 检索质量：

1. **DeepSeek 智能清洗**：通过 ~1100 字的系统提示词引导，保留所有技术细节，消除打印噪声，表格修复，章节锚点拼接
2. **内容保真度校验**：清洗后与原文做相似度对比（`SequenceMatcher`），低于 95% 自动回退
3. **检索关键词注入**：在原始同步脚本中，`add_retrieval_hints()` 向每个章节注入来源文件、文档名称、章节标题作为检索锚点（当前版本已通过 `strip_retrieval_hints()` 剥离，改为依赖章节标题升级）
4. **Dify 分段自定义**：deepseek 管线使用 `\n## ` 作为分段分隔符，确保每个 Chunk 是一个独立的语义单元
5. **重排序优化**：Dify 工作流启用 qwen3-rerank 重排序模型，提升检索相关性

## 11. 微信通道

微信通道是 CloudRAG-Hub 的移动端入口，通过 iLink / ClawBot 协议与微信互通。

### 11.1 技术链路

```
微信用户 ↔ 微信服务器 ↔ iLink API ↔ CloudRAG 后端（轮询）
```

1. 后端通过 `ILinkWeixinClient` 主动长轮询 iLink 接口获取新消息
2. 获取到消息后经过标准化、去重处理
3. 文本消息委托给 `handle_weixin_event` 处理（搜索知识库文件或转发到 Dify 问答）
4. 回复通过 iLink 发送文本/文件接口回传给微信用户

### 11.2 消息类型处理

| 消息类型 | 处理逻辑 |
|----------|----------|
| **文本** | 先模糊匹配知识库文件 → 有匹配则返回文件选项列表 → 无匹配则转发 Dify 获取 AI 答案 |
| **文件** | 下载到本地 knowledge/ 目录 → 确认上传成功 → 同步脚本会后续推送到 Dify |
| **打开文件** | 根据文件名在知识库中查找 → 找到则将文件加密上传到微信 CDN → 发送给用户 |

### 11.3 文件传输安全

微信通道的文件传输使用 AES-ECB 加密：
- **上传**：文件内容经过 AES-ECB 加密后上传到微信 CDN，返回加密参数和 AES 密钥
- **下载**：从微信 CDN 下载加密文件，使用 AES-ECB 解密后保存到本地

加密函数实现在 `services/channels.py` 中的 `_aes_encrypt()` 和 `_aes_decrypt()` 辅助函数。
